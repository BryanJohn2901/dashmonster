# src/app/api/tracking — convenções desta pasta

- Captura (`events_log`) **não depende** de `meta_pixel_id`/`meta_capi_token` — grava sempre que a empresa existe e o domínio bate. Repasse pra Meta CAPI é best-effort, só roda se ambos os campos estiverem preenchidos (senão `capi_status` nasce `"skipped"`, não `"pending"`). Decisão deliberada: testar/usar o pixel próprio não pode depender de ter credenciais Meta reais — não reintroduzir o gate 400 antigo sem entender o motivo.
- Identificação do visitante: `pixel.js` gera um UUID e grava em cookie 1ª parte (`_dm_uid`, 400 dias, `SameSite=Lax`) na primeira visita, manda como `user_id` em todo evento. `track-event/route.ts` usa esse `user_id` como `fingerprint_id` quando presente; só cai pro hash `sha256(ip+UA)` (fraco, sem persistência) quando o pixel não manda (cliente antigo em cache, cookies bloqueados). Não remover esse fallback — é rede de segurança, não o caminho principal.
- `pixel.js` dispara **PageView automaticamente** no `Tracker.init()` (sem precisar de ação do visitante) e **Lead** no submit de qualquer formulário — captura email/telefone (hash + texto puro) e também os demais campos nomeados em `pii.fields` (texto puro, ver `extra_fields` abaixo). WhatsApp/`dataLayer`(Purchase) foram removidos da v2 do pixel — reduzir escopo pra simplificar debug; se forem reintroduzidos, replicar o padrão de `send()` já existente.
- **Deduplicação Pixel (browser) + Conversions API (server), v4**: `send()` gera um `event_id` (UUID) por evento e manda ele tanto pro nosso `/track-event` (que repassa em `capiPayload.data[].event_id`) quanto pro `fbq('track', eventName, {}, {eventID})` no navegador — é a chave que a Meta usa pra reconhecer os dois como o mesmo evento e não contar em dobro. **Não gerar `event_id` diferente entre as duas chamadas** — é o erro mais comum que quebra a dedup (ver `developers.facebook.com/.../deduplicate-pixel-and-server-events`).
  - O fbq da Meta só é carregado no navegador se `meta_pixel_id` estiver configurado (`GET /api/tracking/config?client_id=slug`, endpoint público que devolve só o Pixel ID — nunca o token, nunca o domínio — Pixel IDs já são públicos em qualquer site que usa Meta Pixel). Decisão confirmada com o usuário: o site dele só tem o nosso `pixel.js`, nenhum Meta Pixel manual instalado em paralelo — **não injetar o fbq automaticamente se isso puder colidir com um Pixel já instalado manualmente pelo cliente**; se isso virar um caso real, vai precisar de um toggle "já tenho o fbq instalado" na config.
  - `event_id` é gerado por evento (`send()`), diferente do `user_id`/`_dm_uid` (persistente, por visitante) e do `_fbp`/`_fbc` (por navegador/clique) — os 3 IDs têm escopos diferentes, não confundir.
  - `events_log.event_id` (migration 036) guarda esse ID só pra cross-check manual no nosso dashboard contra o Diagnóstico/Eventos de teste da Meta — não é usado pra nenhuma lógica de dedup do nosso lado, a dedup é 100% feita pela Meta.
- **fbp/fbc** (`_fbp`/`_fbc`, cookies 1ª parte da própria Meta): `getFbp()` só lê o cookie `_fbp` (criado pelo fbq quando carrega). `getFbc()` lê `_fbc` ou, se ausente, reconstrói a partir de `?fbclid=` na URL atual (formato `fb.1.<timestamp>.<fbclid>`) e grava o cookie — **nunca inventar `fbc` sem um `fbclid` real**, isso piora o Event Match Quality em vez de melhorar. Mandados como string crua pro `user_data` da CAPI (não são PII, não hasheiam).
- `companies.meta_test_event_code` (migration 036) — código opcional da aba "Eventos de teste" do Events Manager. Quando preenchido, todo evento dessa empresa inclui `test_event_code` no payload da CAPI (top-level, ao lado de `data`), aparecendo em tempo real lá. Por orientação da própria Meta, deve ficar em branco em produção (só usar durante validação manual).
- `events_log.lead_email`/`lead_phone` guardam email/telefone em **texto puro** (decisão explícita do usuário, v3) — distintos de `user_data.em`/`ph` (hash SHA-256, é o que vai pra Meta CAPI). `pii: {email, phone}` no payload do pixel é só pra esses dois campos — **nunca** misturar no `capiPayload.user_data` da chamada à Meta (tem que continuar só hash). Mesmo padrão de PII em claro que `public.leads` já usa (`028_multi_source.sql`), sem encriptação adicional, só RLS.
- `events_log.extra_fields` (JSONB, migration 033) guarda **todos os outros campos nomeados** do `<form>` além de email/telefone — `pii.fields: {name: value}` no payload do pixel, capturado em `attachFormListener` varrendo `input/select/textarea` (exclui `submit/button/hidden/password/file/reset/image`, limite de 25 campos, valor truncado em 500 chars). Mesma regra de PII em claro de `lead_email`/`lead_phone`: nunca repassado à Meta CAPI, só exibição no dashboard.
- `events_log.page_title` (migration 033) guarda `document.title` capturado pelo pixel em cada evento — usado no dashboard pra mostrar o nome real da página em vez de só a URL/slug. Enviado em todo `send()`, não só no Lead.
- `events_log.country`/`country_region`/`city` (migration 034) — geolocalização por evento, substitui a necessidade de um VisitorAPI externo. Vem de `geolocation(request)` de `@vercel/functions`, que só lê os headers `x-vercel-ip-*` que a rede da Vercel já injeta em toda requisição — **zero chamada de API externa, zero custo, zero latência extra, IP do visitante não é mandado pra terceiros**. Só funciona em produção na Vercel; em dev local os 3 campos vêm `null` (não tem como simular sem deployar). Não trocar por uma API de geo-IP paga sem entender esse trade-off.
- `insertEvent()` em `track-event/route.ts` generaliza o fallback de coluna-ainda-não-migrada: `OPTIONAL_COLUMN_GROUPS` lista os grupos de colunas adicionados por cada migration recente (033, 034, 036); se o insert falha citando uma coluna de um grupo, remove o grupo inteiro e tenta de novo. `selectCompany()` faz o mesmo pro SELECT em `companies` (`meta_test_event_code`, migration 036). Deploy de código (`git push`) é automático mas migration é manual no Supabase SQL Editor — sempre existe uma janela onde o código já espera uma coluna que o banco ainda não tem, e isso não pode derrubar a captura. Ao adicionar uma migration nova com colunas opcionais, adicionar o grupo/fallback aqui também (e no `EVENTS_SELECT`/`EVENTS_SELECT_FALLBACK` de `TrackingEventsView.tsx` e `fetchCompanyTracking()`/`setCompanyTracking()` de `useCompany.ts`, que têm o mesmo problema do lado da leitura/escrita do client).
- CORS em `track-event` é dinâmico (ecoa o `Origin` da request) porque `companies.dominio_autorizado` é por empresa. `Access-Control-Allow-Origin` só aceita 1 valor estático por resposta — a validação real de domínio acontece na lógica da rota (comparação de hostname), não na camada CORS do browser.
- `dominio_autorizado` guarda só o **hostname**, sem porta nem protocolo (ex: `localhost`, não `localhost:3000` nem `http://localhost:3000`) — `new URL(origin).hostname` nunca inclui porta. Coberto por teste em `__tests__/tracking.test.ts`.
- `companies.meta_capi_token` é distinto de `companies.meta_access_token` (este último é o token de gestão de anúncios já existente nas rotas `src/app/api/meta/*` — não serve pra Conversions API).
- Falha na chamada à Meta CAPI nunca pode virar 500 pro pixel — o pixel sempre recebe resposta rápida (200) independente do resultado do POST pra Meta, pra não travar o submit do form do cliente.
- `pixel.js/route.ts` serve o script via route handler (não `public/pixel.js`) pra poder injetar a origem da API em runtime sem passo de build.
- Quem pode editar `meta_pixel_id`/`meta_capi_token`/`dominio_autorizado`/`meta_test_event_code` (via `setCompanyTracking()`/`TrackingConfigPanel`): owner OU manager (migrations `035_tracking_manager_write.sql` + `036_tracking_capi_quality.sql`, que adiciona a 4ª coluna na whitelist). RLS de `companies` é `can_write_company` (owner+manager) pra UPDATE em geral, mas um trigger `BEFORE UPDATE` (`check_companies_update_scope`) restringe manager a só essas 4 colunas — qualquer outra coluna de `companies` continua owner-only. Se adicionar uma coluna nova em `companies` que manager também deveria poder editar, precisa adicionar ela na whitelist `allowed_keys` do trigger (a migration já documenta o porquê do whitelist em vez de blacklist).
- `src/app/api/tracking/config/route.ts` (GET, CORS aberto, sem auth) — endpoint público só pra `pixel.js` descobrir o `meta_pixel_id` da empresa em runtime (pra decidir se carrega o fbq). Devolve só `{ metaPixelId }`, nunca `meta_capi_token`/`dominio_autorizado`/`meta_test_event_code` — esses ficam só no servidor.
